1️⃣ App Store & Play Store Review Justification

(This is written specifically to pass review smoothly)

You can paste this into App Review Notes (Apple) and Policy Declarations (Google).

App Name

StillHere

App Purpose

StillHere is a voluntary safety check-in app designed for people living independently, their families, and caregivers. It provides reassurance through scheduled check-ins and trusted contacts — not continuous monitoring.

Location Usage Explanation

StillHere requests location permission only if the user explicitly enables it.

Location sharing is OFF by default

Location is used only during emergencies or work shifts, based on user settings

Users can turn location sharing off at any time

Location data is not collected continuously

No long-term location history is stored by default

Location is used solely to help trusted contacts find the user during a safety event.

Background Activity

StillHere does not run constant background tracking.
Any live location sharing is time-limited, user-initiated, and clearly indicated in the UI.

Emergency & Safety Behavior

Emergency contacts are notified before any escalation

Contacts can confirm they are checking on the user

StillHere does not automatically contact police or emergency services in the current version

This design minimizes false alarms and respects user autonomy.

Privacy Commitment

StillHere does not sell or share user data

No advertising or tracking SDKs

No audio, video, or message monitoring

Minimal data collection by design

Summary

StillHere is a privacy-first safety app focused on consent, clarity, and human-first escalation. It complies with App Store and Play Store safety and privacy guidelines.

2️⃣ Abuse Response Playbook

(This protects you legally, operationally, and reputationally)

This is internal, but written simply so you can actually follow it.

Abuse Scenarios & Responses
A) OTP / Login Abuse

Risk: repeated OTP requests, SMS spam, cost abuse
Prevention:

OTP cooldown (≥60s)

Daily OTP limits per phone

IP + device throttling

Response:

Temporarily block OTP for the number

Log event

Notify user only if necessary

B) Contact Alert Misuse

Risk: someone spams emergency contacts
Prevention:

No anonymous alerts

Human confirmation + delay

Contact responsibility flow

Response:

Pause alerts

Review logs

Suspend account if malicious

Revoke contact tokens if needed

C) False Emergency Panic

Risk: elderly user forgets phone / accidental SOS
Prevention:

Countdown before escalation

Contact-first design

Response:

Contact confirms safety → incident resolved

No emergency services triggered

Log resolution

D) Lost Phone / SIM Swap

Risk: unauthorized access
Response:

OTP re-verification required

Session rotation

User can revoke contact access links

E) Data Breach (unlikely by design)

Why impact is limited:

No continuous tracking

No long-term location history

Minimal stored personal data

Response:

Notify affected users

Rotate tokens

Publish transparent update

Golden Rule

We respond calmly, clearly, and human-first. No automation panic.
